RV Datavis: Proof of Concept

Guidance notes

These notes provide RV with guidance to use the Proof of Concept (PoC) data visualisations.

All three visualisations delivered in as a web page/application. Data (in .tsv text filesData used are raw data tables taken from the Victorian Waste Performance and Waste Projection Model and VLGAS release notes. An additional table of data (by year) for any miscellaneous metrics (if/where used) in the graphics is also ingested.) is ingested into and then rendered directly in the web browser, using the native web technology stack/languages (i.e. HTML/CSS/JS and SVG).

A web application is ideally suited to either connecting to a live database, or manually updating the application with new/latest data to update the visualisation (e.g. annually). To demonstrate this flexibility, all three visualisations have been programmed wth a 'year' selection input that will update the visualisation with historical data that is included in the ingested data tables.

It should be noted that the scope of the PoC was to provide static visualisations, accordingly:

  • Limited documentation/guidance is provided for updating the data sources: this is not onerous, but may require some training and instruction documentation, in conjunction with an assessment of existing RV tools and capabilities.
  • Limited interactivity, illustrationIt should be noted that there may be brand considerations with the style of illustration (note: EPA use a 'line drawing' style in their brand book, for infographics). Accordingly, as illustration is not in scope for the PoC, those provided can be freely used 'as is', but will not undergo refinement until a further production stage. and animation/transitions has been included where suitable. These are not technically 'in scope' been included 'free' to give each visualisation a more polished feel.
  • As the visualisations are made in browser, they can be natively embedded in websites. The only requirement for this is that the application is hosted 'somewhere'Like all web applications, the files must be hosted on a server to run in a web browser.. To assist RV and to ensure that graphics are easily shareable and socialised, the project code repository has been connected to a development server.

1. Static export of visualisations

The core use case for the PoC graphics is for use as static graphics. To provide RV wit some flexibility, each graphic can be viewed (and therefore exported) with or without its header/sidebar section, i.e. as a 'visOnly' option. This may be preferred for use cases such as inclusion in slides, where different headers and annotation may be adapted.

i. Vector image (pdf or svg)

Each visualisation is rendered in SVG (the data graphic) and HTML/CSS (for adorning text/headers and components like buttons). These are 'publication quality' vector graphic formats (as opposed to bitmap images) that are: infinitely scalable/do not blur on zoom; much smaller in file size; are fully editable in graphic design software. The most natural export format for web vector graphics is as a PDF, which can be achieved by simply using the web browsers Print > to PDF functionality.

By default, each graphic is set to print (in A4) to as either:

  • a landscape page, with adorning text/components located on the left: this can be considered as a 'presentation' layout, or;
  • a portrait page, with adorning text/components located top and bottom: this may be more useful for print publication.

The outputted PDF files Are with graphic design software (i.e. editable if being added/tweaked for a publication) and can be dropped into software like MS Word/Powerpoint.

ii. Conversion to bitmap formats

In cases where a bitmap image (e.g. .png, .jpg etc.) is the only option, the PDF can be convertedAn application is required for conversion from PDF to bitmap formats. While these are freely available, RV will need to install appropriate conversion software. or a screenshot taken.

2. Website embedding

The most logical way for graphics to be embedded on a Vic Gov website using an "iframe". This is the same approach taken by software such as PowerBI. Configuration of the iframe simply involves setting attributes of height and width, and are expected to be done by RV (i.e. internal digital services).

In general, only a 'link' to the web graphic should need to be required (e.g. one of those provided to in the development server). However, as it is unknown 'where' or in what context each graphic might be embedded, or whether/what server would be used, no further guidance can be provided as part of the PoC.

Updating data

As alluded to above, all data visualisattions are powered by data tables that are included as text files (.tsv format) that can be found in the ./_shared/data/ folder in the code repository. For convenience, each graphic load all filesTheoretically, optimisation could be performed, however this is non-trivial work for minimal gain, and a single DataModel is considered far more robust for a PoC., and then transforms them using a shared DataModel class that makes all data series and aggregations accessible for use with each individual graphics visualisation code (which is encapsulated in its own DataVis class).

The two primary tables used in the graphics are directly taken from the latest release notes for the Waste Performance and Projection Model, and the VLGAS data. These are simply exported from their Microsoft Excel spreadsheet format, into a text file (in tab separated values formatIn production this may be tested and/or changed to .csv if preferred. Tab separated is used as its generally 'safer' and more robust, as it can handle numbers formatted with commas, and/or text comment fields that might include commas). The update simply requires these files to be updated and replaced on the server.

The third table is a 'custom' table that features any metres — such as the circularity metrics or economic/valuation metrics — that might be included in any visualisation (e.g. the 'system' data graphic). These metrics have been manually transposed from the Circular Economy Market Report. As its unclear if there is a structured and publicly available data source for these metrics, and/or whether these metrics will be included in the 'final' graphics, the update process for this table is left undocumented and would be considered in 'production' (i.e. beyond the PoC)Note: this is not considered onerous, but would be 'manual'. It is also likely that any 'year-specific' annotation, if used, may be merged into this table and process.

Expansion to interactive data stories

The use of web-based data graphics provides an ideal basis for creating more in-depth, interactive data visualisations and 'stories'. While these are all far beyond the PoC scope, a (very) brief list of possibilities is included below, for documentation completeness:

  • Tooltips (details on hover): providing details of (unlabelled) data values when hovering/touching data representations.
  • Highlighting (on hover/click): highlighting parts of the graphic that related to what is being interactive with. For example, in the system graphic, tapping on a material to see only the flows/values related to that material. ANd in the 'journey' graphic, this may be used to show data that couldn't be included/fitted into the graphic.
  • Toggled views:: a user interface toggle to select or filter a view, e.g. switching all volumes from absolute to per capita values.
  • Staged annotation (and highlighting): this has the most potential in 'system' visualisation, where an interface may be created to step users through different parts of the system. The great benefit here is that more detail can be provided as separate layers of annotation, without over-crowding the graphic itself. For the 'journey' visualissation, this approach could be adapted to mark areas of interest (i.e. spikes/trends in the small charts) that could be expanded upon in a callout box etc.
  • Animation: this can be coupled with any of the ideas above, and can also include animation of the illustrative components.
